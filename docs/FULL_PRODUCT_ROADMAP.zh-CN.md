# ProjectPack Office Agent：完全体产品路线图

> 本文定义 MVP 之后的产品目标、分期交付和协作边界。它不是“让模型自由操作项目”的路线图；所有会改变正式项目数据、通知他人或连接外部系统的动作，都必须有可追溯证据、明确权限和人工确认。

## 1. 完全体长什么样

完全体是面向小型至中型团队的“持续项目办公室”：它持续汇集项目资料，识别候选任务和风险，基于可回溯证据核验进展，并将**待人确认**的建议送到正确的人，而不是只在一次上传后生成一份静态报告。

```text
资料库 / 会议纪要 / 任务系统 / 代码与测试结果
                     │
                     ▼
        受控导入、版本管理与项目知识库
                     │
                     ▼
  候选任务 / 证据检索 / 规则核验 / 模型解释
                     │
                     ▼
   项目看板、风险中心、报告中心、待确认中心
                     │
           人工确认 / 审批 / 评论
                     │
                     ▼
  经授权后才同步任务系统、发送通知或生成正式产物
```

用户在一个项目页面中应当能够：

1. 看见资料何时更新、哪些文件解析失败、索引是否最新；
2. 看见任务、负责人、截止日期、依赖、当前状态和状态变化原因；
3. 点开任一结论，回到原文文件、页码、标题、Sheet 或单元格；
4. 处理“待确认”事项、风险、候选任务和报告草案；
5. 审批后导出或同步结果，而不是让模型直接改原始任务；
6. 查看每次运行、模型版本、提示词版本、耗时、失败原因和审计记录。

## 2. 产品边界与不变原则

### 2.1 允许自动完成的事

- 导入、解析、增量建索引和检索；
- 根据已确认规则生成状态建议、风险建议和报告草案；
- 在项目内创建新的派生产物、运行记录和审计日志；
- 在得到用户明确触发后执行一次受控报告运行。

### 2.2 必须人工确认的事

- 创建、修改、关闭或删除正式任务；
- 覆盖用户上传的源文件；
- 对外发送邮件、IM 消息或同步第三方任务系统；
- 关闭高风险事项、确认“已完成”、变更负责人/截止日期；
- 使用项目外资料作为结论依据。

### 2.3 全阶段不变的技术约束

- 项目隔离：所有资料、向量索引、报告和日志按 `project_id` 隔离；
- 证据优先：无证据时输出“待确认/未找到证据”，不把猜测写为事实；
- 工具白名单：模型不能直接执行 Shell、任意 Python、任意 URL 或任意文件路径；
- 可回滚：每个写操作有操作人、前后值、原因、时间和 `run_id`；
- 可测量：每个阶段都有固定数据集、功能测试、失败测试和云端验证证据。

## 3. 分期总览

| 阶段 | 名称 | 核心结果 | 是否影响正式数据 |
| --- | --- | --- | --- |
| E | 稳定化与可观测性 | MVP 从演示工具变为可重复使用服务 | 否 |
| F | 任务生命周期与人工确认 | 任务不再依赖每次上传 CSV | 是，必须确认 |
| G | 持续知识与风险监控 | 资料变化后自动发现影响和风险 | 否，默认只建议 |
| H | 团队协作工作台 | 项目成员能分工处理风险、报告和确认 | 是，权限控制 |
| I | 外部集成与受控自动化 | 接入任务系统、通知与日历 | 是，审批后执行 |
| J | 生产化与 AMD 优化 | 稳定并发、低显存占用、性能证据 | 否 |

推荐顺序是 **E → F → G → H → I → J**。不要跳过 E 和 F 直接接第三方平台；否则问题无法追溯，且错误同步的代价很高。

## 4. 阶段 E：稳定化与可观测性

### 目标

把当前一次同步运行的 MVP，变成可恢复、可诊断、可复验的服务基础。

### 同事开发清单

- [ ] 将长时间的报告运行改为后台任务：`queued → running → completed/failed`；
- [ ] 前端轮询或服务端推送运行步骤、百分比、当前文件和错误摘要；
- [ ] 为每个步骤记录耗时：解析、Embedding、索引、检索、规则、模型生成、写文件；
- [ ] 支持失败重试和取消，且禁止重复执行同一 `run_id`；
- [ ] 为上传文件增加大小限制、MIME/后缀双重校验、病毒扫描接口预留；
- [ ] 定期清理过期的烟测项目、临时上传和旧向量索引；
- [ ] 统一错误码：配置错误、模型不可达、解析失败、无任务表、无证据、权限不足；
- [ ] 添加运行历史页和单次运行详情页。

### 关键接口

```text
POST /api/projects/{id}/runs              创建后台运行
POST /api/projects/{id}/runs/{run_id}/cancel
GET  /api/projects/{id}/runs/{run_id}     状态、步骤、耗时、错误
GET  /api/projects/{id}/runs              历史列表
```

### 验收

- 30 秒以上的模型调用不会让浏览器请求超时；
- 同一运行失败后可重试，审计日志完整；
- 解析失败的单文件不会毁掉整个项目；
- 页面能明确显示“模型服务不可达”而非泛化报错；
- 云端连续执行 10 次最小演示，无遗留运行状态或数据越界。

### 你的复核重点

- 队列/后台任务不可让模型获得执行任意命令的能力；
- 任何状态更新必须原子化，重启后不能丢失运行；
- 验证取消、重复点击、模型超时、磁盘写满四类失败场景。

## 5. 阶段 F：任务生命周期与人工确认

### 目标

让任务表从“每次上传的静态 CSV”升级为项目内可管理的正式任务库，同时始终保留源任务文件和人工决定权。

### 同事开发清单

- [ ] 设计 SQLite 任务表、任务事件表、确认请求表和操作审计表；
- [ ] 支持创建/编辑任务：标题、负责人、截止日期、优先级、验收标准、依赖关系；
- [ ] 从会议纪要、需求和报告中提取**候选任务**，标记来源和置信度；
- [ ] 候选任务必须在 UI 中由人“接受/修改/忽略”，才写入正式任务库；
- [ ] 把 CSV/XLSX 导入改为“预览差异 → 人工确认 → 导入”，支持重复导入去重；
- [ ] 为任务状态变更建立状态机和历史：禁止从 `cancelled` 自动回到 `completed`；
- [ ] 支持“待确认”队列：谁确认、确认依据、确认备注都必须记录；
- [ ] 报告运行优先读取任务库，同时保留 CSV/XLSX 兼容导入。

### 数据契约

```text
Task: id, project_id, title, owner, due_date, priority, acceptance_criteria,
      source_type, source_reference, version, created_at, updated_at

TaskChange: task_id, before, after, reason, actor, run_id, created_at

Confirmation: subject_type, subject_id, proposed_action, evidence_ids,
              requested_by, decided_by, decision, comment, created_at
```

### 验收

- 模型只能生成候选任务，不能直接创建正式任务；
- 用户能从源文件查看候选任务提取依据；
- 导入同一任务表两次不会产生重复任务；
- 状态和负责人变化都可回溯；
- 生成报告时能区分“系统建议”与“人工确认后的正式状态”。

### 你的复核重点

- 先冻结 Task / Confirmation Schema，再允许 UI 和抽取模块并行开发；
- 检查是否存在绕过确认直接写 SQLite 的路径；
- 用冲突数据测试：资料称完成、任务负责人称未完成时必须进入待确认。

## 6. 阶段 G：持续知识与风险监控

### 目标

从一次性“生成报告”升级为资料发生变化时主动提示项目影响。

### 同事开发清单

- [ ] 资料版本管理：文件 SHA-256、解析版本、索引版本、替换关系；
- [ ] 增量索引：只重算变更文件及相关 Chunk；
- [ ] 建立风险规则库：临近截止、逾期、无证据、验收缺失、依赖阻塞、资料冲突；
- [ ] 风险去重、聚合、严重程度和生命周期；
- [ ] “变更影响分析”：资料更新后列出可能受影响的任务和报告；
- [ ] 定时扫描任务与资料，但默认只创建风险建议，不外发通知；
- [ ] 检索质量基准集：事实题、跨文档题、无答案题、冲突题；
- [ ] 记录召回率、引用正确率、拒答率、耗时和失败率。

### 验收

- 修改一份资料后，只有相关索引和相关风险重新计算；
- 已关闭风险不会因同一证据重复出现；
- 无答案题必须保持拒答，不得输出编造引用；
- 质量指标可按模型、Embedding 模型、索引版本进行对比。

### 你的复核重点

- 风险规则必须可配置、可测试，不能把判断全部藏在 Prompt；
- 固定并版本化评测数据，优化前后必须对比；
- 定时任务不得绕开项目隔离或读到其他项目索引。

## 7. 阶段 H：团队协作工作台

### 目标

让团队成员在同一界面处理项目，而不只是管理员下载报告。

### 同事开发清单

- [ ] 登录与项目级角色：管理员、项目经理、成员、只读访客；
- [ ] 项目总览：进展、风险、待确认、最近资料变更和最近运行；
- [ ] 任务看板/列表：筛选、排序、负责人视图、截止日期视图；
- [ ] 风险中心：指派负责人、评论、处理状态、确认关闭；
- [ ] 报告中心：草案版本、人工编辑、审批、导出 PDF/DOCX；
- [ ] 评论与提及：讨论必须挂在任务、风险或报告段落上；
- [ ] 通知收件箱：站内通知优先，不直接接入外部 IM；
- [ ] 文件权限与下载权限：成员只能访问有权限的项目。

### 验收

- 不同项目成员不能枚举或下载其他项目的资料/报告；
- 报告草案可编辑，但原始 AI 产物和人工编辑版本均可追溯；
- 高风险关闭需要有权限的人员确认；
- 关键操作写入不可修改的审计记录。

### 你的复核重点

- 把鉴权放在 API/服务层，而非只隐藏 UI 按钮；
- 检查对象级权限：项目、文件、任务、风险、报告都必须验证；
- 先实现最小角色模型，不要在此阶段自建复杂组织系统。

## 8. 阶段 I：外部集成与受控自动化

### 目标

在人工授权的前提下，把已确认的结果同步给现有工作系统。

### 同事开发清单

- [ ] 定义 Connector 抽象：读取、预览差异、申请确认、执行、回滚、审计；
- [ ] 优先接入一种任务系统（例如 GitHub Issues 或 Jira），不要同时接多个；
- [ ] 支持从外部任务系统只读同步任务和状态；
- [ ] 支持“生成同步预览”：列出将创建/修改的字段和理由；
- [ ] 用户确认后才执行写回，并记录外部对象 ID；
- [ ] 接入一种通知渠道（邮件或 Teams/Slack），仅发送已确认的摘要；
- [ ] Token 使用服务端密钥存储，不出现在日志、报告或浏览器；
- [ ] 失败重试、限流、幂等键和死信处理。

### 验收

- 未确认的候选任务和风险不能写回外部系统；
- 同一同步请求重复提交不会创建重复 Issue；
- 外部 API 失败有明确重试状态和人工处理入口；
- 撤销/回滚范围可见，且不会删除用户的原始外部数据。

### 你的复核重点

- 每个 Connector 单独一个 S2/S3 Spec 和单独 PR；
- Secret 管理、权限最小化和审计优先于功能速度；
- 对外写入必须有“预览 + 确认 + 结果”三步证据。

## 9. 阶段 J：生产化与 AMD Radeon 优化

### 目标

让系统在真实负载下稳定运行，并形成可提交的 AMD Radeon 性能证据。

### 同事开发清单

- [ ] 任务队列并发控制、每项目配额、模型调用排队和取消；
- [ ] 将聊天模型与 Embedding 模型拆分：选择更小的专用 Embedding 模型，避免双 35B 常驻；
- [ ] 对索引、Embedding 和报告结果实施缓存与失效策略；
- [ ] 大文件、批量文件、长上下文和多项目并发压测；
- [ ] 记录 ROCm、GPU 型号、llama.cpp 版本、量化格式、上下文、GPU Layers；
- [ ] 采集首 Token 延迟、生成 Tokens/s、Embedding 吞吐、端到端耗时、VRAM、GPU 利用率；
- [ ] 对比优化前后结果，固定基准数据和测量方法；
- [ ] 备份、恢复、日志轮换、健康检查、告警与容量阈值。

### 验收

- 在目标 Radeon GPU 上重复运行固定基准，性能数据可复现；
- 高并发时不会因显存不足导致全部运行失败；
- 模型/服务失效时 API 和 UI 给出可恢复提示；
- 不同项目之间无数据串读，旧数据可按保留策略清理。

### 你的复核重点

- 不以“GPU 100%”作为性能结论；必须记录吞吐、时延和端到端耗时；
- 优化前后使用同一数据、同一模型版本和同一上下文参数；
- 性能改动不能破坏引用正确率、拒答率和安全边界。

## 10. 两人协作方式

### 同事：应用开发负责人

- 负责各阶段的业务模块、UI、测试夹具和对应 Spec；
- 一次 PR 只做一个可验收主题；
- 交付时提供：输入样例、接口契约、测试命令、结果、已知限制；
- 不修改云端模型参数、密钥、`main` 保护规则或共享 Schema，除非先由你确认。

### 你：管理员 / 技术负责人 / 复核人

- 冻结公共 Schema、项目隔离规则、确认语义和外部写入边界；
- 审核 PR 的 Spec、数据迁移、权限、安全、回滚与性能证据；
- 在云端运行集成验证，维护模型服务与基准环境；
- 只在 required checks 和复核清单通过后合并；
- 对每阶段决定“进入下一阶段”或“补测试/补安全修复”。

## 11. 每个阶段的 PR 门禁

同事每个 PR 必须附带：

```md
## Scope
- Stage:
- Spec ID:
- User-visible change:

## Contracts and data
- New/changed schema:
- Migration and rollback:
- Permission/confirmation impact:

## Verification
- Unit/integration commands and result:
- Failure cases covered:
- Cloud/performance evidence, if required:

## Risk
- Known limitation:
- Rollback plan:
```

你合并前至少检查：

- [ ] PR 标题、Spec、测试报告和 required checks 均通过；
- [ ] 没有 `.env`、模型、真实用户资料、Token、向量库或运行产物；
- [ ] 输入输出和公共 Schema 兼容，或有版本/迁移说明；
- [ ] 任何写入、同步、删除都有权限、确认、审计和回滚；
- [ ] 涉及 RAG/模型/性能的改动有云端或可复现集成验证；
- [ ] PR 范围没有夹带无关重构。

## 12. 下一步：从阶段 E 开始

下一次开发不要直接做第三方集成。第一个完全体 PR 应是：

```text
feat(runs): add durable background run lifecycle
```

它应先完成后台运行状态、取消/重试、步骤耗时、运行历史和失败可见性。这会成为之后任务确认、风险监控、协作和外部集成共同依赖的基础。

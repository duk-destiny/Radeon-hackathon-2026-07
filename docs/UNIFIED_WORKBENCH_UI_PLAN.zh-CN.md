# ProjectPack Office Agent：统一工作台 UI 实施方案

> 目的：把已经存在的项目、资料、运行、任务、风险、报告、协作和受控集成能力，组织成一个可供团队日常使用的统一界面。本方案不改变“证据优先、人工确认、项目隔离、服务端鉴权”的产品边界。

## 1. 目标形态

当前 Gradio 页面已完成 MVP 演示闭环：创建项目、上传资料与任务表、生成报告、查看结果、下载 Markdown/CSV/下周计划。

完全体 UI 不是把所有按钮堆到一个页面，而是让用户以“项目”为入口完成以下工作：

```text
登录
  └─ 项目列表 / 项目总览
       ├─ 资料库：上传、解析状态、版本、索引状态
       ├─ 运行中心：启动、进度、取消、历史、失败原因、产物
       ├─ 任务中心：看板、导入预览、候选任务、人工确认、变更历史
       ├─ 风险中心：筛选、负责人、讨论、关闭确认、证据
       ├─ 报告中心：草稿、编辑、审批、导出、来源引用
       ├─ 协作中心：成员、评论、@ 提及、通知
       └─ 集成中心：同步预览、人工确认、执行结果、审计
```

任何“写入正式任务、关闭风险、审批报告、发送通知、同步第三方系统”的动作，都必须在界面上明确展示影响范围、证据和确认人；UI 不能绕过 API 的权限与确认校验。

## 2. 技术路线与边界

### 2.1 建议路线

- **保留 Gradio**：作为比赛演示和最小闭环入口，不在其上继续堆叠完整协作后台。
- **新增统一 Web 工作台**：建议 `web/` 独立目录，使用 React + TypeScript + Vite；通过 FastAPI 的 `/api/...` 接口访问后端。
- **后端不搬家**：项目、任务、风险、报告、认证、审计和队列规则仍留在现有 Python 服务层；前端只渲染数据、提交用户操作和展示错误。
- **同源部署优先**：生产环境由 FastAPI/反向代理托管编译后的前端，避免浏览器跨域、Token 泄漏和环境地址不一致；开发环境可由 Vite 代理 `/api`。

这项 UI 开发可在没有云端 GPU 的情况下进行：使用本地 API、固定演示数据和 Mock Service Worker 测试。云端只负责最后的真实模型/RAG、性能和并发验收。

### 2.2 不能由 UI 决定的事情

- 不在浏览器保存模型地址、云端 SSH 密钥、外部系统 Token 或任意文件路径；
- 不以隐藏按钮代替权限校验；服务端仍必须拒绝无权限请求；
- 不让前端直接调用 llama-server，也不让模型从页面执行任意工具；
- 不把“模型建议”显示为“已确认事实”；无证据必须保留“待确认/证据不足”；
- 不在一个 PR 内同时重写前端、后端 Schema 和 RAG 流程。

## 3. 现有能力与 UI 对接范围

| 领域 | 当前已有后端能力 | UI 要补的体验 |
| --- | --- | --- |
| 项目与资料 | 项目创建、文件上传、受控路径、解析/RAG 流程 | 项目列表、资料状态、失败原因、版本与索引状态 |
| 运行与产物 | 创建/执行/取消/查询运行；报告、风险 CSV、下周计划下载 | 实时进度、步骤耗时、历史、重试、可读的错误和产物预览 |
| 任务 | CSV/XLSX 导入、候选任务、确认队列、生命周期、审计 | 任务表格/看板、导入差异预览、确认操作、历史抽屉 |
| 风险 | 风险扫描、负责人、状态和变更影响分析 | 风险列表、筛选、证据展开、负责人处理与关闭确认 |
| 报告 | 草稿、审批、PDF/DOCX 导出 | Markdown 编辑器、版本对比、审批流、引用跳转 |
| 协作 | JWT、角色、成员、评论、通知 | 登录、成员页、评论区、通知收件箱、权限提示 |
| 集成 | 邮件/SCM 等预览、确认、执行和审计能力 | 集成配置状态、差异预览、确认对话框、执行记录 |
| 运维 | 队列、缓存、监控、备份、压力测试接口 | 管理员健康视图、队列状态、失败提示；不向普通成员暴露管理操作 |

开始每个页面前，开发者必须用 API 契约测试确认该接口的路径、认证方式、成功体和错误体；表中“已有能力”不等于所有细节都已经适合直接展示。

## 4. 分期与交付顺序

每一期单独一个功能分支、一个 Spec、一个 PR。前一期的接口契约和验收通过后，再开始后一期。

### UI-0：基础与契约冻结（先做）

**目标**：搭好可持续开发的前端骨架，确认前后端接口，而不是先画静态页面。

**开发清单**：

- [ ] 创建 `web/`、TypeScript、路由、构建和开发代理；不改现有 Gradio 页面。
- [ ] 定义统一 API Client：`baseUrl`、Bearer Token、超时、网络错误、`401/403/404/409/422/5xx` 错误映射。
- [ ] 为项目、运行、任务、风险、报告、认证接口写 TypeScript DTO；不得手写与 Pydantic 不一致的字段名。
- [ ] 建立应用壳：登录态恢复、顶部导航、项目选择器、加载/空状态/错误页。
- [ ] 建立 Mock 数据和接口契约测试；没有云端时使用它们完成 UI 回归。
- [ ] 输出 `docs/UI_API_CONTRACT.md`：逐项列出已验证接口、缺失接口、字段不一致和负责人。

**验收**：未登录访问受保护路由会跳转登录；网络异常有可理解提示；前端构建、静态检查、关键 API Client 测试全部通过。

**不得做**：不改公共 Schema，不将 Token 写入日志或 Git，不用 Mock 伪造“模型已经运行”。

### UI-1：项目总览、资料库与运行中心

**用户场景**：项目经理新建“客户交付 A”，上传会议纪要、需求文档和任务表，点击生成报告后能持续看到当前步骤、失败原因和历史产物。

**开发清单**：

- [ ] 项目列表：创建、选择、项目摘要、最近运行和权限提示。
- [ ] 项目总览：资料数量、任务状态统计、开放风险、待确认数量、最近运行。
- [ ] 资料库：上传、列表、解析成功/失败、错误摘要、文件元数据；仅显示当前项目资料。
- [ ] 运行中心：创建运行、轮询状态、当前步骤、耗时、取消、重试入口、历史运行。
- [ ] 产物区：在线预览 Markdown，下载报告/风险 CSV/下周计划；下载链接必须走受控 API。
- [ ] 统一错误提示：模型不可达、文件不支持、无任务、证据不足、运行失败分别说明下一步。

**验收**：用演示资料完成“项目创建 → 上传 → 运行 → 查看任务证据 → 下载三类产物”；刷新页面后仍可从历史运行找到产物。

**依赖/缺口处理**：若后端没有资料版本或索引状态查询接口，只在 UI-1 的 Spec 中列为缺口，不以扫描服务器目录的方式绕过后端。

### UI-2：任务工作台与人工确认

**用户场景**：成员从任务 CSV 导入任务，系统从会议纪要抽取候选项；项目经理查看差异和证据后，确认哪些内容可以成为正式任务。

**开发清单**：

- [ ] 任务列表与看板：按状态、负责人、优先级、截止日筛选和排序。
- [ ] 任务详情抽屉：状态、验收标准、来源证据、变更历史、关联风险。
- [ ] CSV/XLSX 导入预览：显示新增/更新/重复/无效行，确认前不写入正式任务。
- [ ] 候选任务与确认队列：接受、修改、忽略；展示提取证据与确认原因。
- [ ] 生命周期操作：只向有权限用户展示可用转换；服务端拒绝时显示具体原因。
- [ ] 审计视图：显示谁、何时、为何更改了字段或状态。

**验收**：重复导入不重复建任务；成员无法确认管理员操作；任意任务状态均可回看来源和历史。

### UI-3：风险中心与报告中心

**用户场景**：负责人查看临近截止、证据不足的风险，分配处理人并讨论；报告作者编辑草稿，项目经理审批后导出正式文件。

**开发清单**：

- [ ] 风险列表、严重程度/状态/负责人筛选、证据与影响范围展开。
- [ ] 风险处理动作：指派、评论、建议关闭、确认关闭；高风险关闭需要明确确认弹窗。
- [ ] 报告中心：草稿列表、版本时间线、Markdown 编辑与预览、来源引用链接。
- [ ] 审批界面：提交、批准、驳回、要求修改，保留意见和审批历史。
- [ ] 导出入口：PDF/DOCX；失败时展示可恢复的错误。

**验收**：普通成员不能审批；关闭风险和批准报告均留下审计记录；引用不能跳转到其他项目文件。

### UI-4：协作与通知

**用户场景**：两人团队中，管理员给同事分配项目权限；同事在风险或报告段落上评论并 @ 管理员，管理员在收件箱看到可定位的通知。

**开发清单**：

- [ ] 登录、退出、会话过期处理和角色可见性。
- [ ] 成员管理：邀请、角色展示、移除；仅管理员可操作。
- [ ] 任务/风险/报告段落评论，@ 提及和跳转。
- [ ] 站内通知收件箱：未读、已读、按项目筛选、跳转目标实体。
- [ ] 权限拒绝页面与审计提示。

**验收**：跨项目访问被 API 拒绝且 UI 不泄露名称/内容；评论和通知可追溯到具体对象。

### UI-5：受控集成与管理员运维视图

**用户场景**：管理员把已确认任务同步到外部系统，先看到字段差异预览，再确认执行；同时可查看模型、队列和系统健康，不直接暴露密钥。

**开发清单**：

- [ ] 集成中心：连接状态、只读同步结果、待执行变更列表。
- [ ] 同步三步流：预览差异 → 人工确认 → 执行结果/可回滚信息。
- [ ] 外部通知预览和确认；绝不在页面显示 Secret 或完整 Token。
- [ ] 管理员运维页：健康、队列长度、缓存摘要、最近失败、备份状态。
- [ ] 危险操作二次确认、角色校验、操作审计；普通用户无入口也无 API 权限。

**验收**：未确认不会写外部系统；重复确认不会创建重复对象；非管理员访问运维接口得到 `403`。

### UI-6：可用性、性能与发布

**开发清单**：

- [ ] 响应式布局、键盘可达性、中文错误文案、长文本/大表格性能优化。
- [ ] 路由级懒加载、分页、筛选条件 URL 化、请求取消与防重复点击。
- [ ] 前端错误监控（脱敏）、版本号、构建产物校验和回滚说明。
- [ ] Playwright 端到端演示脚本：登录 → 上传 → 运行 → 确认任务 → 审批报告。
- [ ] 云端恢复后，补真实 RAG 运行、并发队列、浏览器长时间运行的验收证据。

**验收**：在固定演示数据和真实云端各完成一轮端到端流程；不以 GPU 利用率代替端到端性能结论。

## 5. 给同事的开发规则

### 分支与 PR

```text
feat/ui-foundation
feat/ui-project-runs
feat/ui-task-confirmation
feat/ui-risk-reports
feat/ui-collaboration
feat/ui-integrations-admin
```

- 每个分支只实现本方案中的一个 UI 阶段或一个明确子主题。
- 每个实现 PR 在 `specs/ui-<topic>-YYMMDD-HHMM/` 下提供 `PRODUCT.md`、`TECH.md`；S2/S3 完成后必须有 `TEST_REPORT.md`。
- PR 标题继续使用官方固定格式：`Track 2, PLASMA, ProjectPack Office Agent`。
- 不能把 `.env`、用户资料、模型文件、索引、输出、Token 或截图中的隐私数据提交进仓库。

### 每个 PR 必须交付

```md
## Scope
- UI stage:
- Spec ID:
- User-visible change:

## API contract
- Endpoint(s) used or added:
- Auth/role requirement:
- Error states covered:

## Verification
- Unit/component test command and result:
- Browser/E2E command and result:
- Mock or cloud environment used:

## Risk and rollback
- Known limitation:
- Rollback:
```

### 复核清单（管理员）

- [ ] 页面没有绕过服务端权限、确认或项目隔离。
- [ ] 所有加载、空数据、无权限、网络失败、模型失败和提交中状态都可理解。
- [ ] 前端 DTO 与后端 Schema 有契约测试；没有靠猜字段拼页面。
- [ ] 写操作防重复提交，并能显示服务端返回的审计/执行结果。
- [ ] 变更不包含密钥、真实资料、模型、索引或运行产物。
- [ ] `Governance` required checks、Spec 校验和测试通过。

## 6. 建议的首三个 PR

1. **UI-0 / `feat/ui-foundation`**：`web/` 基础、登录态、路由、API Client、错误处理、Mock 和 API 契约文档。
2. **UI-1a / `feat/ui-project-runs`**：项目列表、项目总览、运行历史、产物下载；不包含任务看板。
3. **UI-1b / `feat/ui-files-progress`**：资料上传、解析状态、运行进度和失败提示；只复用 UI-1a 的项目上下文。

这样同事可以先交付一个可访问、可测且不依赖云端的 UI 基础；你负责复核 API 契约、权限和确认边界。云端恢复后，再集中做真实 RAG 与 AMD 性能验收，不阻塞页面开发。

# ProjectPack Office Agent：应用开发负责人工作计划

## 1. 角色定位

你负责把“项目资料”变成可信的办公结果：**解析资料、建立检索、匹配任务证据、生成带来源的周报与风险清单，并完成最小 Gradio 工作台**。

你不负责云端账号、模型部署、`main` 合并、密钥或比赛最终提交；这些由管理员/技术负责人负责。

## 2. 你的开发边界

你可以修改：

```text
app/rag/        文件解析、Chunk、索引、检索、引用
app/tools/      文件、任务、RAG、报告工具
app/reports/    周报、风险清单、下周计划
app/ui/         Gradio 最小工作台
tests/          解析、检索、规则、报告测试
specs/          当前任务对应 Spec 与验收报告
```

你不能修改，除非先与管理员确认：

```text
llama-server 启动参数
云端模型路径和 ROCm 配置
.env 中的密钥或真实地址
项目根目录/数据落点安全边界
GitHub main 规则
任务与证据的公共 Pydantic Schema
```

## 3. 阶段任务

### 阶段 A：资料导入与解析

目标：让系统可靠地读取演示项目资料，并知道每段文字从哪里来。

- [ ] 支持 Markdown、TXT、PDF、DOCX、XLSX。
- [ ] 扫描 `data/projects/<project_id>/source/`，拒绝符号链接逃逸和项目目录外路径。
- [ ] 为每个文件记录：相对路径、格式、文件 SHA-256、大小、修改时间、解析状态、错误信息。
- [ ] PDF 保留页码；DOCX 保留标题/段落；Excel 保留 Sheet、表头和单元格范围。
- [ ] 解析异常不能中断整个项目导入；应将坏文件列为失败项。

**验收标准**：用同一套演示资料导入，能列出成功/失败文件，且引用可定位到原文件、页码/章节/Sheet。

### 阶段 B：RAG 与带来源问答

目标：对项目资料提问时，回答必须有来源；没有证据时明确拒答。

- [ ] 实现按标题优先的 Chunk 切分：500–1000 中文字符、100–150 重叠、表格独立保留。
- [ ] 使用文件哈希和解析版本做增量索引；源文件未变化时不得重复构建。
- [ ] 实现 FAISS 向量召回与 BM25 关键词召回，并合并后取 Top 4–6。
- [ ] 每个检索结果返回文件名、相对路径、位置、摘要和评分。
- [ ] 生成答案时强制输出来源；检索证据不足时返回“未在项目资料中找到证据”。

**验收标准**：完成 20 个固定问题集：10 个事实题、5 个跨文件题、5 个无答案题；记录命中、引用正确和拒答结果。

### 阶段 C：任务核验与报告

目标：把“上周任务表”与项目资料证据对应，输出可解释的状态和报告草稿。

- [ ] 从 XLSX/CSV 读取任务：标题、负责人、截止日、优先级、验收标准、原始来源。
- [ ] 为每项任务检索证据；先用规则检查时间、成果、完成关键词、验收标准和缺失项。
- [ ] 将任务、证据、规则结果交给模型，仅让模型输出结构化解释和风险原因。
- [ ] 状态至少支持：已完成、基本完成、进行中、未开始、已延期、待确认、已取消。
- [ ] 生成 Markdown 周报和 CSV 风险清单；每项状态和风险都附来源或明确证据缺口。
- [ ] 生成下周计划草稿，不得直接覆盖用户原任务表。

**验收标准**：演示资料中“系统测试”只有计划和部分记录、没有最终报告时，系统必须判为“待确认/进行中”，不能误判完成。

### 阶段 D：最小工作台

目标：让评委一次演示看懂完整闭环。

- [ ] 一个 Gradio 工作台即可：项目选择、资料导入状态、输入任务、执行步骤、来源、报告下载、GPU 状态。
- [ ] 不先做五个独立页面、登录、多人协作或复杂审批。
- [ ] UI 只调用管理员提供的受控后端接口，不直接访问任意文件路径或模型进程。

**验收标准**：从导入到生成报告不需要终端操作；错误会显示可理解的提示。

## 4. 开发方式

### 每次开始

```powershell
cd "D:\AI module\heikesong"
git switch main
git pull --ff-only origin main
git switch -c feat/<short-topic>
```

先判断等级：S0 只在 PR 中写目标和验证；S1 创建简单 Spec；S2/S3 创建完整 Spec、验证矩阵和验收报告。

### 每次结束

```powershell
git diff --check
py scripts/validate_specs.py --strict
py -m unittest discover -s tests -p "test_*.py"
git status
git add <本次相关文件>
git commit -m "feat(rag): add source citation retrieval [rag-YYMMDD-HHMM]"
git push -u origin feat/<short-topic>
```

不要使用 `git add .`；不要把模型、`.env`、资料原件、索引、输出或日志加入暂存区。

## 5. PR 要求

PR 标题必须是英文比赛格式：

```text
Track 2, <真实队名>, ProjectPack Office Agent
```

PR 描述必须说明：

- 本次改变了什么、为什么；
- Spec ID 或 `S0/no-spec` 原因；
- 输入样例与输出样例；
- 已执行命令和结果；
- 风险、未覆盖情形和回滚方式；
- 是否需要管理员进行云端模型验证。

等待以下检查通过后，再请管理员合并：

```text
PR title / official submission format
Hygiene and specifications
```

## 6. 开发原则

- 先保证来源、规则和错误提示，再追求“像人一样的 Agent”。
- 模型只能解释检索到的证据，不能凭空补全任务状态。
- 任何写入都只能落在 `outputs/`；原始资料默认只读。
- 没有证据时写“待确认”或“未找到证据”，不是猜测。
- 每次 PR 只完成一个可验收主题；解析、检索、报告、UI 不混成一个大 PR。
- 向管理员报告阻塞，而不是私自更改云端、模型参数或公共 Schema。
